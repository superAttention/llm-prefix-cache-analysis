01 / Abstract

The narrow question

The core question is narrower than generic Belady framing: on a ShareGPT-style chat trace, how large is the gap between common online prefix-cache eviction policies and an upper bound under the same leaf-only candidate constraint?

The current repository answers that question under a deliberately clean model. Each cached radix node stores exactly one fixed-size block. The reported benchmark uses page_size = 1, so one block equals one token for accounting. In plain terms, this is a diagnostic baseline, not a production-faithful page configuration, but it preserves the explicit block-eviction semantics needed for a well-defined oracle.

The key result is large: lru reaches a peak relative gap of 80.2% versus tc_belady at 1271 blocks. The main failure mode is not random noise. It is a systematic tendency to keep freshly touched deep tail blocks that are already dead, which motivated the depth_lru heuristic.

Research scope

Not a full production simulator of all SGLang features. A report-oriented eviction study under a fixed-block, page-aligned radix model with a tree-constrained offline oracle and a shadow-oracle mechanism analysis.

The safe reading: this measures an upper bound under the same leaf-only candidate constraint, not an unconstrained global optimum.

relative_gap = (tc_belady − baseline) / tc_belady
02 / Trace

Trace construction

The input data is a cached local snapshot of ShareGPT_Vicuna_unfiltered. Each conversation is processed in order. Every human turn becomes one access, and the accessed object is the entire accumulated prompt prefix up to that turn.

This matters because the cache is not serving isolated utterances. It is serving prefixes that grow over time. A later request can hit a long shared prefix even when the final tail diverges.

Across conversations, accesses are globally interleaved rather than concatenated sequentially. The resulting benchmark trace contains 300 conversations and 978 accesses.

Tokenizer. The benchmark uses Meta-Llama-3-8B-Alternate-Tokenizer. The serialized trace remains token-level; blocking is applied later inside the simulator.
Arrival model. Interleaving order is random with seed 0 — a modeling choice, not a measured production arrival schedule.
03 / Model

Fixed-block cache model

The simulator uses a fixed-size block radix tree. Each node stores exactly one block. Prefix matching only counts on full block boundaries. A trailing partial block is still a block and still consumes one unit of cache capacity.

In the current benchmark, page_size = 1. That removes internal fragmentation from the reported run while keeping the explicit block semantics that make the offline oracle tractable.

Accounting rule

Benefit is measured in reused tokens and reused full requests. Cost is measured in blocks. Under the current run, the full-cache point is 319,862 blocks, which matches the number of unique block prefixes in the trace.

  • one radix node per block
  • block budget, not token budget
  • leaf-only online eviction
  • full-block prefix hits only
  • same tree for online and offline
  • explicit candidate leaf set
04 / Policies

The simulator supports standard online policies, one heuristic, and one offline upper bound. All online policies operate on the same block radix tree and choose from the same current leaf set.

Priority policy note

In SGLang, request metadata can carry a priority value that is propagated into the radix cache. The local simulator implements the same rule, but the current ShareGPT trace carries no request-priority metadata. Every node receives priority 0, so priority reduces to plain lru in this benchmark.

05 / Oracle

Tree-constrained offline oracle

The offline comparator remains named tc_belady, but under the current design it is the offline optimum for the simplified fixed-block model, not for unconstrained token caching in the abstract.

The critical restriction is that online and offline policies share the same object model: one node per block, one leaf-block eviction unit, and the same candidate pool at each eviction event. The oracle evicts the leaf block whose block-aligned prefix is reused farthest in the future.

victim = argmaxleaf next_access(prefix)

In report terms, this is an upper bound under the same leaf-only candidate constraint. It does not measure a broader optimum that could restructure the tree or release non-leaf state early.

06 / Mechanism

Shadow-oracle mechanism analysis

The mechanism analysis does not run two independent simulations and compare their final trees. That would confound the result with divergent state trajectories. Instead, it uses a shadow-oracle design.

One online strategy drives the real simulation. At each eviction event, the code inspects the exact same current leaf set and asks what tc_belady would have evicted from that candidate pool.

|A| = |B| = number of disagreement events

The invariant holds because each disagreement contributes exactly one online-only victim and one oracle-only victim.

A · online evicts, oracle retains
B · oracle evicts, online retains
C · both evict same leaf
D · both retain leaf

Observed LRU failure mode

At the LRU peak-gap point, many Group B leaves are both very recent and never reused again. In the current report summary, Group B for LRU has median time_since_last_access = 0, time_to_next_access = ∞, and tree_depth = 1077. That is the signature of false warmth: deep conversation-tail blocks that were just touched but are already dead.

07 / Heuristic

Why DepthLRU exists

Plain LRU knows only recency. It does not distinguish between a recently touched broadly shared prefix and a recently touched deep tail block near the end of a conversation. In this workload, that distinction matters.

A deep block can look extremely warm because it was touched by the latest human turn, yet still be dead immediately afterward. Meanwhile, a shallower prefix can look slightly older while remaining reusable across later accesses.

priority = last_access_index − α · depth

Current design: α = 0.01 · page_size. The reported page_size = 1 benchmark therefore uses α = 0.01.

Interpretation

depth_lru is a structural correction to recency, not a new semantic signal. Under this trace construction it may also be acting as a proxy for "chat tail position," which is why the implementation scales the depth penalty with page_size and why robustness beyond page_size = 1 still matters.

09 / Fidelity

Alignment & intentional simplifications

Aligned pieces

  • page-aligned block semantics
  • radix-tree prefix matching at block boundaries
  • leaf-based candidate pool for eviction
  • priority-aware eviction rule shape
  • per-node recency & frequency metadata

Explicit simplifications

  • fixed-size block model (not full allocator)
  • no lock-state modeling
  • no extra-key namespaces
  • no Eagle / bigram mode
  • report benchmark uses page_size = 1

A precise explanation of the current repository state — not a claim that every production SGLang detail has been simulated.
