Abstract

The core question is straightforward: given a realistic multi-turn chat workload, how much headroom exists between common online prefix-cache eviction policies and an offline upper bound that sees the future?

The current repository answers that question under a deliberately clean model. Each cached radix node stores exactly one fixed-size block. The reported benchmark uses page_size = 1, so one block equals one token for accounting, while still preserving the explicit block-eviction semantics needed for a well-defined oracle.

The key result is large: lru reaches a peak relative gap of 80.2% versus tc_belady at 1271 blocks. The main failure mode is not random noise. It is a systematic tendency to keep freshly touched deep tail blocks that are already dead, which motivated the depth_lru heuristic.

Research Scope

This is not a full production simulator of all SGLang features. It is a report-oriented eviction study under a fixed-block, page-aligned radix model with a tree-constrained offline oracle and a shadow-oracle mechanism analysis.

relative_gap = (tc_belady - baseline) / tc_belady

Trace Construction

The input data is a cached local snapshot of anon8231489123/ShareGPT_Vicuna_unfiltered. Each conversation is processed in order. Every human turn becomes one access, and the accessed object is the entire accumulated prompt prefix up to that turn.

This matters because the cache is not serving isolated utterances. It is serving prefixes that grow over time. That means a later request can hit a long shared prefix even when the final tail diverges.

Across conversations, accesses are globally interleaved rather than concatenated sequentially. The resulting benchmark trace contains 300 conversations and 978 accesses.

Tokenizer. The current benchmark uses NousResearch/Meta-Llama-3-8B-Alternate-Tokenizer. The serialized trace remains token-level. Blocking is applied later inside the simulator.

Fixed-Block Cache Model

The simulator uses a fixed-size block radix tree. Each node stores exactly one block. Prefix matching only counts on full block boundaries. A trailing partial block is still a block and still consumes one unit of cache capacity.

In the current benchmark, page_size = 1. That removes internal fragmentation from the reported run while keeping the explicit block semantics that make the offline oracle tractable.

Accounting Rule

Benefit is measured in reused tokens and reused full requests. Cost is measured in blocks. Under the current run, the full-cache point is 319,862 blocks, which matches the number of unique block prefixes in the trace.

  • one radix node per block
  • block budget rather than token budget
  • leaf-only online eviction
  • full-block prefix hits only
  • same tree model for online and offline policies
  • explicit candidate leaf set at eviction time

Eviction Policies Under Comparison

The simulator supports standard online policies, one heuristic, and one offline upper bound. All online policies operate on the same block radix tree and choose from the same current leaf set.

Priority Policy Note

In SGLang, request metadata can carry a priority value that is propagated into the radix cache. The local simulator implements the same rule, but the current ShareGPT trace carries no request-priority metadata. As a result, every node receives priority 0, so priority reduces to plain lru in this benchmark.

Tree-Constrained Offline Oracle

The offline comparator remains named tc_belady, but under the current design it is the offline optimum for the simplified fixed-block model, not for unconstrained token caching in the abstract.

The critical restriction is that online and offline policies share the same object model: one node per block, one leaf-block eviction unit, and the same candidate pool at each eviction event. The oracle evicts the leaf block whose block-aligned prefix is reused farthest in the future.

victim = argmax_leaf next_access(prefix)

This fixed-block design is what made the oracle clean. Earlier compressed-leaf designs muddied the eviction unit because a single leaf could represent multiple tokens with different future reuse structure.

Shadow-Oracle Mechanism Analysis

The mechanism analysis does not run two independent simulations and compare their final trees. That would confound the result with divergent state trajectories. Instead, it uses a shadow-oracle design.

One online strategy drives the real simulation. At each eviction event, the code inspects the exact same current leaf set and asks what tc_belady would have evicted from that candidate pool.

|A| == |B| == number of disagreement events

This invariant holds because each disagreement contributes exactly one online-only victim and one oracle-only victim.

A: online strategy evicts, oracle retains
B: oracle evicts, online strategy retains
C: both evict the same leaf
D: both retain the leaf

Observed LRU Failure Mode

At the LRU peak-gap point, many Group B leaves are both very recent and never reused again. In the current report summary, Group B for LRU has median time_since_last_access = 0, time_to_next_access = inf, and tree_depth = 1077. That is the signature of false warmth: deep conversation-tail blocks that were just touched but are already dead.

Why DepthLRU Exists

Plain LRU knows only recency. It does not distinguish between a recently touched broadly shared prefix and a recently touched deep tail block near the end of a conversation. In this workload, that distinction matters.

A deep block can look extremely warm because it was touched by the latest human turn, yet still be dead immediately afterward. Meanwhile, a shallower prefix can look slightly older while remaining reusable across later accesses.

last_access_index - alpha * depth

The heuristic does not try to predict the future directly. It only penalizes depth so that recently touched deep leaves compete against slightly older shallower leaves instead of automatically surviving every time.

Interpretation

depth_lru is a structural correction to recency, not a new semantic signal. It is motivated by the observation that the missing information in LRU is partly about where a leaf sits in the evolving prompt tree.

Main Read

The heuristic helps, but only slightly. depth_lru improves over lru at every non-full cache size, yet the peak gap only moves from 80.2% to 79.3%. That means depth captures part of the failure mode, but not enough to close the gap materially.

Runtime. The six-size benchmark sweep recorded 353.70s real, 331.60s user, and 13.15s sys.

Alignment With SGLang and Intentional Simplifications

Aligned Pieces

  • page-aligned block semantics
  • radix-tree prefix matching at block boundaries
  • leaf-based candidate pool for eviction
  • priority-aware eviction rule shape
  • per-node recency/frequency style metadata for online policies

Explicit Simplifications

  • fixed-size block model rather than full allocator behavior
  • no lock-state modeling
  • no extra-key namespaces
  • no Eagle/bigram mode
  • report benchmark uses page_size = 1

This makes the page a precise explanation of the current repository state, not a claim that every production SGLang detail has been simulated.
